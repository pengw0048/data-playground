# Measured local performance and memory envelope — issue #640

## Scope and identity

- Product commit measured: `3b5fc99229ac144762f1feccfff7149cfc9b49a8` (`test: de-flake the ux-acceptance full matrix (#653)`), the released engine on `main`. The same SHA is recorded in [envelope.json](envelope.json) under `machine.git_sha`.
- Harness: [`scripts/perf_envelope.py`](../../../scripts/perf_envelope.py) — a self-contained, seeded, headless certification of a small set of named supported local workflows. It drives the real in-process engine (`Deps` → `compiler.compile_plan` → `LocalRunner.run` / `profile_node`), not a mock.
- Reference run: macOS 26 / arm64, 16 logical CPUs, 64 GiB RAM, CPython 3.12.13, DuckDB 1.5.4. The committed [envelope.json](envelope.json) is that reference run's raw output; the scheduled CI tier regenerates it on Linux/x86-64 and uploads the fresh envelope as a build artifact.
- This is a bounded certification leaf: three workflows plus one out-of-memory boundary case, one script, one evidence doc. It is not a benchmark platform and adds no product behavior, no per-PR gate, and no config field.

## What the envelope characterizes

The memory boundary is the **configured DuckDB budget** (`DP_MEMORY_LIMIT`, the product's per-kernel RAM cap). DuckDB's own default budget is ~80% of physical RAM, so a physical-RAM shortfall degrades the same way; pinning the budget makes the envelope deterministic and machine-independent for the contract assertions (spill happens, the write commits, the boundary fails truthfully) while the raw timings/RSS stay machine-specific reference values.

## Named workflows

All four share one shape — `source → rollup (group-by) → managed-local write` — at two sizes, plus a profile pass and the same oversized rollup with the spill volume bounded.

| Workflow | Dataset | Budget | Spill cap | Certifies |
| --- | --- | --- | --- | --- |
| `under_budget_write` | `orders_small` (200k rows, CSV) | 2 GB | — | Normal managed-local write, well under budget. |
| `over_budget_spill` | `orders_large` (6M rows, Parquet) | 256 MB | — | Same shape exceeding the budget: DuckDB spills to disk and the run still **completes correctly**. |
| `profile_over_budget` | `orders_large` | 256 MB | — | One full profile pass over **every** row of the larger dataset. |
| `oom_boundary` | `orders_large` | 256 MB | 32 MB | Spill volume also bounded: the run **fails with a truthful error and commits nothing**. This is the workflow #489 ("measured workflow") builds on. |

## Reference envelope

From [envelope.json](envelope.json) (`2026-07-20`, reference machine above):

| Workflow | Threads | Outcome | Rows | Peak spill | Peak RSS | Wall |
| --- | --- | --- | --- | --- | --- | --- |
| `under_budget_write` | 16 | `done`, 1 committed, output verified | 200,000 | 0 MiB | 226 MiB | 0.16 s |
| `over_budget_spill` | 2 | `done`, 1 committed, output verified | 3,000,000 | 369 MiB | 452 MiB | 0.64 s |
| `profile_over_budget` | 2 | `done`, full (unsampled) | 6,000,000 scanned | 0 MiB | 192 MiB | 0.15 s |
| `oom_boundary` | 2 | `failed`, 0 committed | — | 18 MiB (hit the 32 MiB cap) | 512 MiB | 0.24 s |

Thread counts follow the product's existing rule (`_apply_session` lowers threads to `DP_MEMORY_LIMIT / DP_MIN_MEM_PER_THREAD_MB`), so at a 256 MB budget the large workflows run at 2 threads on any machine with ≥ 2 cores; the 2 GB budget keeps all cores.

## The honest out-of-memory contract

The envelope asserts the boundary from both sides:

1. **Under the budget, over-sized working sets degrade gracefully.** `over_budget_spill` builds a rollup hash table far larger than the 256 MB budget; DuckDB spills ~369 MiB to `DP_SPILL_DIR` and the managed write still commits. `output_verified` re-reads the committed Parquet and confirms `sum(n)` equals the 6,000,000 input rows — a completed run is never a silent wrong result.

2. **Beyond the budget, with the spill volume also exhausted, the run fails truthfully and commits nothing.** `oom_boundary` bounds the spill volume at 32 MiB (`SET max_temp_directory_size`, simulating a full spill disk). The full run path ends `status = failed` with **zero** committed outputs — the runner discards every unpublished partial. The root cause captured on the product's own session config is a DuckDB `OutOfMemoryException` (`"failed to offload data block … 30.4 MiB/30.5 MiB used"`) returning zero rows, proving the failure is the memory boundary rather than an unrelated fault.

**What the user sees:** a run that stops at `failed` with a DuckDB out-of-memory / aborted-transaction error and no new dataset revision — never a truncated or partial result presented as complete.

## Reproducibility

- Datasets are generated by deterministic DuckDB `range()` `COPY`; content checksums (order-independent `bit_xor(hash(*))`) and file byte sizes are stable across runs and machines on the pinned DuckDB 1.5.4 (`orders_small` `646feaab…`, `orders_large` `15f2f5e6…`, recorded in `envelope.json` under `datasets`).
- The outcome, thread count, spill-vs-complete, and truthful-failure contracts reproduce on any machine. Raw timings and peak RSS reproduce within a modest tolerance on the **same** machine; they vary by hardware across machines by design.
- Re-run: `cd kernel && uv run python ../scripts/perf_envelope.py --output ../docs/acceptance/issue-640/envelope.json`. The script exits non-zero if any workflow violates its contract, so the scheduled CI tier fails loudly on a regression.

## Not claimed or exercised

Distributed / Ray measurement, object-store or provider writes, per-PR performance gates, optimization work, comparisons against other tools, or an open-ended benchmark suite. Physical-RAM exhaustion is characterized via the configured DuckDB budget, not by allocating past total system memory.
